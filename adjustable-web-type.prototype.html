Hand-authored Beziers from lib/sculpt.js's glyph registry, tuned by the preset's defaults + glyphParams. The WOFF outline above is rendered for comparison — note that the parametric output below does not derive from it. That mismatch is the architectural problem.

Try the two candidate fixes: Path γ (extended outline-deform: real WOFF + multi-axis global sliders) and Path α (WOFF + per-letter handles, minimal proof on instrumentSerif × {a, o, l}).

Path γ — extended outline-deform (real WOFF + multi-axis sliders)

Starting shape is the WOFF outline. Sliders below deform it globally — every letter responds together. width and serifLength are real path-level deformations (already in lib/sculpt.js). xHeight (vertical scale, baseline-pivoted) and weight (outline dilation via stroke widening) are prototype post-processing on the rendered SVG — production γ would build these into the lib's applyPresetAxesToCommands.

Verdict to bring back: does global multi-axis feel like enough control, or do the letters need per-letter precision?

Path α — WOFF + per-letter anatomy handles (full implementation)

Starting shape is the WOFF outline. Every letter in the wordmark exposes the preset's handle vocabulary at anchor positions derived from each glyph's bounding box:

  • instrumentSerif / bitter: height (top), width (right), serifLength (bottom-right, omitted on round-bottom letters like o/c/e/s/g), weight (left)
  • sourceSans: height, width, weight
  • ibmPlexMono: height, width, weight (note: dragging width breaks the mono cell invariant — that's a known tension to bring back)

Height handle's label adapts per letter: cap-height for uppercase, ascender for b/d/f/h/k/l/t, x-height otherwise. Hover any handle to see which axis on which letter in the tooltip.

Transforms in this prototype: scaleY-base (pivot at baseline), scaleX-left (pivot at glyph left), weight-stroke (per-glyph stroke widening). Production α needs real outline math — region-clipped scale for x-height (scale only above baseline below cap-height), translation of just the serif endpoints, true offset-path dilation for weight.

The height handle's label adapts per letter category: cap-height for uppercase, ascender for b/d/f/h/k/l/t, x-height otherwise. Descenders (g/j/p/q/y) currently scale through a single x-height handle — production α may want a dedicated descenderDepth.

Path α implementation notes for production

  • x-height handle: prototype scales the entire glyph in Y, pivoted at baseline. Production should clip the scaled region to (baseline, x-height) so the ascender/cap-height stays fixed. Requires path-level region splitting.
  • width handle: prototype scales the entire glyph in X, pivoted at left edge. Production should preserve stem thickness (scale only the bowl/counter regions, not the stems). Requires anatomy-aware path partitioning.
  • serifLength handle: prototype scales X (same as width with a different anchor visualization). Production should translate only the serif endpoint anchors along the baseline. Requires identifying serif anchor points on the outline.
  • weight handle: prototype uses SVG stroke widening on the existing fill. Production should perform real outline offset (Minkowski sum) so contrast is preserved.
  • descenderDepth: not exposed yet. Should be a second height-family handle that scales only y > 0 (below baseline).
  • Mono width tension: dragging width on ibmPlexMono breaks the mono cell. Either lock width on mono, or let the user opt out of mono per-letter, or add a wordmark-level mono toggle.

Not in scope here

  • The none preset (parametric-only fallback) — removed from this picker.
  • Any change to lib/sculpt.js. Path γ's xHeight/weight and all of Path α live as SVG post-processing in this prototype only.
